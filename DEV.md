# 开发流程规范

## 强制开发流程

每个非trivial任务必须按顺序执行以下流程。如果需要跳过某一步，必须明确询问用户确认后才能跳过，禁止静默跳过。

### 第一步：头脑风暴（skill: brainstorming）
- 触发条件：任何新功能、修改或创意任务
- 目的：在写任何代码之前，探索用户意图、需求、约束和设计方案
- 跳过条件：用户明确说"跳过头脑风暴"，或任务是一行代码级别的小修复

### 第二步：制定计划（skill: writing-plans）
- 触发条件：头脑风暴完成后
- 目的：将需求拆解为具体的、可验证的实现步骤
- 产出：书面计划文件，需用户审批通过
- 跳过条件：用户明确同意跳过

### 第三步：创建 Git 工作树（skill: using-git-worktrees）
- 触发条件：开始实现之前
- 目的：将开发工作与主分支隔离
- 跳过条件：用户明确说在当前分支上直接开发

### 第四步：测试驱动开发（skill: test-driven-development）
- 触发条件：实现每个功能或修复时
- 循环：红（写失败的测试）→ 绿（写最少代码让测试通过）→ 重构
- 此步骤不可省略，每个代码变更都必须有对应测试
- 跳过条件：用户明确说"跳过 TDD"，并知晓这意味着没有测试覆盖

### 第五步：执行计划（skill: executing-plans 或 subagent-driven-development）
- 触发条件：计划审批通过，准备开始实现
- 独立任务可使用 dispatching-parallel-agents 并行执行
- 遇到测试失败或异常行为时，使用 systematic-debugging 进行系统性排查

### 第六步：代码审查（skill: requesting-code-review）
- 触发条件：实现完成后
- 目的：验证工作成果是否满足需求
- 如果收到审查反馈：使用 receiving-code-review skill 进行技术严谨的评估

### 第七步：验证 (skill: verification-before-completion)
- 触发条件：在宣称工作完成之前
- 规则：证据先于断言，必须展示实际命令输出
- 禁止在没有展示证据的情况下声称"测试通过"

### 第八步：文档维护 (Document Maintenance)
- 触发条件：验证通过后，在完成分支之前
- 目的：确保项目主文档 (DOCS.md, README.md) 同步了最新的功能、变更和知识沉淀
- 操作：检查是否引入新功能/配置，若有则更新 DOCS.md 对应章节
- 跳过条件：用户明确说"不需要更新文档"

### 第九步：完成分支 (skill: finishing-a-development-branch)
- 触发条件：所有验证和文档更新通过后
- 展示选项：合并 / 创建 PR / 清理
- 由用户决定集成方式

## 流程执行规则

1. **禁止静默跳步**：如果准备跳过某一步，必须停下来询问："我准备跳过【步骤名称】，原因是【原因】。可以吗？"
2. **计划前不写代码**：在第二步的计划审批通过之前，不得编写实现代码（第四步中的测试代码除外）
3. **完成前必须出示证据**：第七步是强制性的，必须展示实际的测试输出结果
4. **播报步骤切换**：进入下一步时，简要说明："进入第 N 步：【步骤名称】"
5. **流程违反提醒**：如果用户要求做的事情会打破流程（如"直接写代码"），需要提醒："这将跳过【步骤列表】。确定要继续吗？"
