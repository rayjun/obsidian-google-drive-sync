# 项目文档原则 (DOCS.md)

本文件定义了本项目及基于此规范的所有项目的 **文档维护与生成原则**。

---

## 🏗️ 核心记录原则

1.  **追加记录 (Append-Only)**:
    *   新功能、新架构、新决策应作为扩展内容追加到现有文档末尾。
    *   保留历史上下文，这有助于 AI 了解项目的演进过程。

2.  **存量保护 (Persistence)**:
    *   除非文档中的内容已明确过时（例如：被替换的旧 API 或已移除的模块），否则 **禁止删除** 现有的记录内容。
    *   对于过时的内容，建议使用 `[DEPRECATED]` 标记而非直接删除。

3.  **单一真相源 (SSoT)**:
    *   文档必须与代码实现同步。
    *   在开发流程的 **第八步 (Document Maintenance)** 强制执行同步检查。

---

## 📂 文档分类指南

AI 在执行文档更新时，应按照以下分类在相应位置记录：

-   **架构描述 (Architecture)**: 记录系统核心模块、数据流向和技术选型理由。
-   **知识库 (Knowledge Base)**: 沉淀特定业务逻辑、算法实现、复杂 Bug 的根因分析。
-   **API 参考 (API Reference)**: 记录服务接口、函数签名和核心配置项。
-   **操作指南 (Guides)**: 记录安装、部署、调试和运维的具体命令。

---

## 🤖 AI 文档操作指令

当 AI 进行文档维护时，必须：
1.  首先阅读目标文档的完整内容。
2.  识别需要更新的部分所属的章节（Architecture / Knowledge / API / Guide）。
3.  在对应章节的末尾 **追加** 新内容。
4.  确保新内容包含 **更新日期 (YYYY-MM-DD)**。

---

## 📈 长期演进

随着项目规模增长，如果 `DOCS.md` 本身变得过于庞大：
-   支持在 `DOCS.md` 中建立索引。
-   将具体章节拆分到 `docs/` 目录下的子文件中（如 `docs/architecture.md`）。
